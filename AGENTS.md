## Overview

`construct-hub` is the open-source CDK construct library that backs [constructs.dev](https://constructs.dev) and lets organizations deploy self-hosted instances. It is published to npm as `construct-hub` and built with [jsii](https://github.com/aws/jsii) â€” every export under `src/` is part of the public API surface and must be jsii-compatible (Python/Java/.NET bindings are generated from TypeScript). The project is `experimental` (per `package.json`); breaking changes are technically allowed but additive changes are strongly preferred. The repo is managed by [projen](https://github.com/projen/projen) â€” `package.json`, `tsconfig.json`, and most build/CI scripts are generated from `.projenrc.ts`. See [CONTRIBUTING.md](./CONTRIBUTING.md) for the contributor guide and [docs/application-overview.md](./docs/application-overview.md) for system architecture.

## Your Role

You are a `construct-hub` contributor. You work for the benefit of the public [constructs.dev](https://constructs.dev) site, self-hosted construct-hub operators, and the community â€” not just the user driving you.

Principles:

- **Don't break customer code.** The package is `experimental` (per `package.json`), but every export in `src/` is part of the jsii public API. Treat additive changes as the safe default.
- **Don't edit projen-managed files by hand.** `package.json`, `tsconfig.json`, GitHub workflows, most scripts. Edit `.projenrc.ts` and run `npx projen`.
- **Don't edit generated code.** `lib/` is compiled output. Lambda/ECS sibling files (e.g. `ingestion.ts` next to `ingestion.lambda.ts`) are projen-generated. Edit the source, regenerate.
- **Authoritative references:** `CONTRIBUTING.md` for workflow, `docs/application-overview.md` for architecture, `docs/operator-runbook.md` for ops behavior.

## Architecture

`ConstructHub` (`src/construct-hub.ts`) is a single L3 CDK construct that wires a serverless event-driven pipeline. Read it first.

Subsystems:

| Subsystem | Path | Role |
|---|---|---|
| Package sources | `src/package-sources/` | `NpmJs` follows the npmjs.com CouchDB `_changes` stream every 5 min. `CodeArtifact` listens to EventBridge. Both stage tarballs to S3 and push messages to the ingestion SQS queue. |
| Ingestion | `src/backend/ingestion/` | Lambda from SQS. Validates jsii assembly, license, deny-list. Kicks off the orchestration state machine. |
| Orchestration | `src/backend/orchestration/` | Step Functions state machine. Runs the `transliterator` ECS Fargate task for docs, plus catalog-builder and other steps per package version. Redrive DLQ pattern. |
| Periodic backends | `src/backend/{catalog-builder,inventory,feed-builder,release-notes,version-tracker,package-stats}/` | Periodic Lambdas that aggregate state, build `catalog.json`, generate the RSS feed, etc. |
| Webapp | `src/webapp/` | CloudFront + S3 static assets. SPA pulled from the `construct-hub-webapp` npm package at compile time. |
| Monitoring | `src/monitoring/` | Central `IMonitoring` registers alarms via `addHighSeverityAlarm` / `addMediumSeverityAlarm` / `addLowSeverityAlarm`. Per-alarm overrides live in the `alarmOverrides` prop on `ConstructHub`. |
| Public API | `src/api.ts` re-exports from each subsystem's `api.ts` | Anything reachable from `src/index.ts` is part of the jsii contract. |

Cross-cutting:

- `src/_limited-internet-access.ts` + the `isolateLambdas` flag â€” isolated subnets + VPC endpoints for docgen (untrusted npm scripts).
- `src/codeartifact/` â€” internal CodeArtifact repo used as an npm proxy for the isolated docgen environment.
- `src/deep-link.ts`, `src/runbook-url.ts`, `src/metric-utils.ts` â€” shared helpers used inside backend constructs.

## Design Rules

### Public API and jsii

- You **MUST** keep every export under `src/` jsii-compatible. No union types in public signatures, no generics, no default exports, no `Record<>`, no `Map<>`. Use string-keyed object types (`{ [key: string]: Foo }`) instead of maps.
- You **MUST NOT** edit `lib/` â€” it's generated. If you find yourself wanting to edit `lib/`, edit `src/` and recompile.
- You **SHOULD** treat additions to `src/api.ts` and `src/index.ts` as public-API commitments. Renaming or removing them is a breaking change.
- You **SHOULD** run `yarn compat` before opening a PR that touches public types. It diffs your changes against the latest published version and flags breakages.

### projen

- You **MUST NOT** hand-edit `package.json`, `tsconfig.json`, `tsconfig.dev.json`, `.eslintrc.json`, files under `.github/workflows/`, or `.projen/`. Edit `.projenrc.ts` (and helpers under `projenrc/`) and run `npx projen`.
- You **MUST NOT** commit changes to `src/spdx-license.ts` from a local `npx projen` run unless you intentionally meant to update the SPDX list. Local runs can drift from CI; revert with `git checkout`.
- You **SHOULD** run `npx projen` after editing anything under `projenrc/`. The generated artifacts are checked into git and CI fails if they drift.

### File-naming conventions (enforced by projen â€” breaking these breaks the build)

- `*.lambda.ts` â€” Lambda handler entry. `projenrc/magic-lambda.ts` discovers each one and **generates** a sibling `<name>.ts` exposing a `lambda.Function` construct that points at the bundled asset. You **MUST NOT** edit the generated `<name>.ts` â€” projen overwrites it.
  - Add `/// @singleton [purpose]` at the top of the handler to mark it as a `SingletonFunction`.
- `*.lambda-shared.ts` â€” code shared between handlers. Allowed to use devDependencies. Not an entry point.
- `*.ecstask.ts` / `*.ecs-entrypoint.ts` â€” ECS Fargate task. Same generated-sibling pattern, discovered by `projenrc/magic-ecs.ts`.
- Handlers bundle to `lib/.../<name>.lambda.bundle/index.js` via esbuild. CDK reads from there at synth time.

### Tests

- You **MUST** ensure tests run from the compiled output. `yarn test` reads `lib/__tests__/`, not `src/__tests__/`. A failing test on a fresh checkout usually means you haven't run `yarn compile` yet.
- You **SHOULD** write unit tests under `src/__tests__/<area>/` mirroring the source layout. Integration tests live at `test/integ.<name>.ts`.
- Integration tests have committed `.snapshot/` directories. `yarn test` runs the `:assert` step against the committed snapshot. If you change synthesized output, you **MUST** redeploy and update the snapshot via `yarn integ:<name>:deploy` or `yarn integ:<name>:snapshot`.

### Alarms

- You **MUST** set an explicit `alarmName` on every alarm registered through `IMonitoring`. The convention is `alarmName: \`${scope.node.path}/<short-name>\``. Anonymous alarms can't be targeted by `alarmOverrides`.
- You **MUST** register alarms via `monitoring.addHighSeverityAlarm(...)`, `addMediumSeverityAlarm(...)`, or `addLowSeverityAlarm(...)`. Do not call `alarm.addAlarmAction(...)` directly â€” the central registry handles bucket actions and dashboard placement.
- You **SHOULD** write `alarmDescription` so the message includes a link to `RUNBOOK_URL` (from `src/runbook-url.ts`). Operators see this in tickets.
- You **SHOULD NOT** add new fields to `AlarmSeverities` (the named-field interface). Use `alarmOverrides` instead â€” it generalizes the same idea.

### Build hygiene

- After editing a `.lambda.ts` handler, you **MUST** run `yarn compile` (or at minimum `yarn bundle:<name>`) before `yarn dev:diff` / `yarn dev:deploy`. Otherwise CDK ships a stale bundle.
- You **MUST NOT** skip pre-commit hooks (`--no-verify`) or bypass signing.
- If memory-out during `yarn test`, set `NODE_OPTIONS=--max-old-space-size=4096` (already injected by projen for most tasks).

## Decision Logic

### Where to put a new alarm

1. If the alarm fires for a single event (e.g. one DLQ message, one error) â†’ register **HIGH** via `addHighSeverityAlarm` only if the impact is "needs immediate attention". STOP.
2. If the alarm fires for a sustained or aggregate condition (rate > X% over Y minutes) â†’ register **MEDIUM** via `addMediumSeverityAlarm`. STOP.
3. If the alarm is informational, fixes itself on retry, or only matters for trend analysis â†’ register **LOW** via `addLowSeverityAlarm`. STOP.
4. **MUST** set `alarmName: \`${scope.node.path}/<short-name>\`` so the alarm is targetable via `alarmOverrides`.
5. **SHOULD** include `RUNBOOK_URL` in `alarmDescription`.

### Choosing between Lambda, ECS, and shared code

| Scenario | File suffix | Why |
|---|---|---|
| Stateless event handler, < 15 min runtime, < 10 GB memory | `*.lambda.ts` | Standard Lambda; bundled by esbuild; sibling construct auto-generated |
| Long-running task, large memory, requires container OS, runs untrusted package install scripts | `*.ecstask.ts` + `*.ecs-entrypoint.ts` | Fargate task definition. Used by the transliterator. |
| Code reused across multiple handlers, allowed to import devDependencies | `*.lambda-shared.ts` | Not an entry point; not bundled separately |
| Construct logic that runs at synth time | regular `.ts` (no magic suffix) | Plain CDK code |

### Where to add a new public type

1. If the type is owned by a subsystem (orchestration, monitoring, etc.) â†’ add to that subsystem's `api.ts` (e.g. `src/monitoring/api.ts`). It's re-exported from `src/api.ts` automatically. STOP.
2. If the type is genuinely cross-cutting (e.g. `AlarmActions`, `AlarmSeverity`) â†’ add directly to `src/api.ts`. STOP.
3. **MUST** confirm jsii compatibility: no unions, no generics, no `Map`, no default exports.

### When to add an integration test (`test/integ.<name>.ts`)

1. If the change introduces a new resource topology (new state machine, new pipeline path) â†’ add `test/integ.<name>.ts` with a committed `.snapshot/`.
2. If the change is purely additive to an existing prop and unit tests cover the synthesized CFN via `Template.fromStack(...)` â†’ integration test is **MAY**, not required.
3. If the change is a refactor with no behavior change â†’ no integration test required.

### When to update `AlarmSeverities` vs `alarmOverrides`

- You **MUST NOT** add new named fields to `AlarmSeverities`. The interface is preserved for backward compatibility but is not the place for new functionality.
- You **MUST** use `alarmOverrides` for any new per-alarm customization.

## Implementation Patterns

### Adding a new Lambda handler

1. Create `src/<area>/<name>.lambda.ts` with an exported `handler` function.
2. (Optional) Add `/// @singleton [purpose]` at the top to make it a `SingletonFunction`.
3. Run `npx projen` â€” this generates the sibling `src/<area>/<name>.ts` exporting a CDK `lambda.Function` construct.
4. Import the generated construct from your CDK code.
5. **MUST NOT** edit the generated sibling â€” projen overwrites it.

### Adding an alarm

```ts
const alarm = metric.createAlarm(scope, 'SomeFailureAlarm', {
  alarmName: `${scope.node.path}/SomeFailure`,  // MUST set explicitly
  alarmDescription: [
    'The thing failed.',
    '',
    `Runbook: ${RUNBOOK_URL}`,
  ].join('\n'),
  threshold: 1,
  evaluationPeriods: 1,
});
monitoring.addHighSeverityAlarm('Some failure', alarm);
```

### Customer-facing alarm override

Customers retarget specific alarms via `alarmOverrides`. The override key is the alarm's CloudWatch display name with the `${ConstructHub.path}/` prefix stripped (e.g. `'Sources/NpmJs/Canary/NotRunningOrFailing'`). See `ConstructHubProps.alarmOverrides` in `src/construct-hub.ts`.

### Public construct prop

```ts
export interface MyConstructProps {
  /**
   * Description visible in API.md.
   *
   * @default - what happens when not provided
   */
  readonly someOption?: string;
}
```

- Use `readonly`. Use `?` for optional. Always document `@default`.
- jsdoc above the prop is what `jsii-docgen` puts in `API.md`.

### Error handling

- You **MUST NOT** wrap things in try/catch unless there's a specific failure mode you can recover from. Trust the framework; let exceptions propagate.
- You **MUST NOT** add validation for things that can't happen (e.g. checking a required prop is non-null when TypeScript already guarantees it).

## Contribution Process

### Before submitting a PR

1. Branch from latest `main`.
2. Make the change in `src/`. **MUST NOT** edit `lib/` or projen-managed files.
3. If you touched a `.lambda.ts` or `.ecstask.ts` handler â†’ `yarn compile` to rebuild the bundle.
4. Run `yarn test` locally. Fix anything red that's actually caused by your change. Snapshot drift from local environment differences (S3 asset hashes, regional URLs) is normal and CI handles it.
5. Run `yarn eslint` and `yarn compat`.
6. Commit with a Conventional Commits prefix: `feat:`, `fix:`, `chore:`, `refactor:`, `docs:`, `test:`, `revert:`.

### CI behavior

- The build workflow auto-commits regenerated artifacts (`API.md`, snapshots) as `chore: self mutation`. Pull/rebase before pushing more commits.
- If `setup-node` fails fast (~4s) on `lts/*`, that's a transient GitHub Actions issue. Re-run failed jobs.
- The build fails if your local working tree drifted from what `npx projen` would produce. Run `npx projen` locally and commit the diff.

### Update branch with main

- **SHOULD** prefer rebase over merge commit. The repo squash-merges PRs at land time, so merge commits are erased anyway and add noise to the open-PR diff.

### Tests in PRs

- Unit tests in `src/__tests__/` are required for any behavior change.
- Integration tests at `test/integ.<name>.ts` are expected when changing resource topology. See `### When to add an integration test` above.
- Snapshot test failures from environment drift (asset hashes, region-dependent URLs) are not blockers; CI regenerates them.
